# JTC Excel設計書 Markdown化デモアプリ 要件定義

作成日: 2026-06-15  
対象リポジトリ: `ZennAI-Japan/jtc-excel-md-converter`  
目的: 大手企業に多い Word / Excel 設計書、とくに複数シートに分かれた Excel方眼紙設計書を、AI活用しやすい Markdown / JSON / レビュー可能な形式へ変換するデモアプリの要件を定義する。

---

## 1. 背景と課題

大手企業のシステム開発・保守では、設計書や業務仕様書が Word / Excel 形式で長年蓄積されている。特に Excel 設計書は、単なる表ではなく、複数シートに分かれたブック全体で仕様を表現していることが多い。

例:

- 表紙
- 改訂履歴
- 画面一覧
- 画面設計
- 項目定義
- 入力チェック
- イベント定義
- コード表
- API / DB 項目対応

この形式では、AI検索・RAG・仕様質問・テストケース生成に使う前に、人手で Markdown やテキストへ整理する作業が発生する。現状の課題は以下。

1. Word / Excel から Markdown への変換に時間がかかる。
2. Excel方眼紙では、セル値だけでなく罫線・結合セル・色・コメント・入力規則・座標配置に仕様情報が埋まっている。
3. 複数シートに分散した仕様の関係性が失われやすい。
4. MarkItDown などの汎用変換だけでは、JTC設計書の意味構造を十分に保持できない。
5. 変換結果が正しいか、人間が確認しにくい。

---

## 2. デモアプリの目的

このデモアプリは、単なるファイル変換ツールではなく、以下を示すための営業・PoC向けアプリとする。

> 複数シートに分かれた Excel設計書を、AIが読める統合仕様書へ再構成できる。

初期デモでは、AIチャットやRAG本番連携までは必須にしない。まずは、顧客が見て価値を理解しやすい「ドキュメント化」体験を完成させる。

---

## 3. ターゲットユーザー

### 3.1 主ユーザー

- 大手企業の情報システム部門
- SIer / 開発ベンダーのPM・SE
- 保守開発チーム
- DX推進部門

### 3.2 利用シーン

- 古いExcel設計書をAI検索基盤へ投入したい。
- 保守開発前に既存仕様を読み解きたい。
- 属人化した設計書をレビュー可能なナレッジに変換したい。
- 移行・刷新プロジェクトで既存仕様の棚卸しをしたい。

---

## 4. デモで見せるコア体験

### 4.1 基本フロー

1. ユーザーが Excel設計書ファイルをアップロードする。
2. アプリがブック内の複数シートを解析する。
3. シート一覧と推定カテゴリを表示する。
4. 各シートの抽出結果を Markdown / HTML preview / JSON で確認できる。
5. ブック全体を統合した Markdown 仕様書を出力する。
6. 変換時の warnings と評価レポートを表示する。
7. 変換結果を ZIP または個別ファイルとしてダウンロードできる。

### 4.2 初期デモの価値訴求

- 複数シートのExcel設計書を、ブック単位で整理できる。
- Markdown化により、人間にもAIにも読みやすくなる。
- HTML preview により、元Excelのセル座標・表構造を確認できる。
- warnings により、変換が不確かな箇所を隠さずレビューできる。
- evaluation により、抽出できた構造量を説明できる。

---

## 5. 機能要件

### 5.1 ファイルアップロード

- ユーザーは `.xlsx` ファイルを1つアップロードできる。
- 初期デモでは `.xls` / `.xlsm` / `.docx` / `.pdf` は対象外とする。
- ファイルサイズ上限は初期値として 20MB を想定する。
- アップロード後、ファイル名・シート数・変換ステータスを表示する。

### 5.2 複数シート解析

アプリは Excel ブック内の全シートを解析する。

各シートについて以下を抽出する。

- シート名
- 表示順
- 結合セル見出し
- 罫線ブロック
- セル値
- セル座標
- コメント
- 入力規則・プルダウン候補
- 塗りつぶし色
- warnings

初期デモでは、非表示シート・図形・画像・マクロ・外部リンクの完全抽出は対象外とし、検出できた場合は warnings に出す。

### 5.3 シート分類

シート名と内容から、簡易的にシート種別を推定する。

候補:

- 表紙
- 改訂履歴
- 画面一覧
- 画面設計
- 項目定義
- 入力チェック
- イベント定義
- コード表
- その他

初期デモでは決定論的なルールベースでよい。LLM分類は次フェーズとする。

### 5.4 ブック概要生成

変換結果の先頭に、ブック全体の概要を生成する。

含める内容:

- ファイル名
- シート数
- シート一覧
- 推定シート分類
- 検出した罫線ブロック数
- 検出した入力規則数
- warnings数

### 5.5 Markdown出力

以下のMarkdownを出力する。

1. `book_specification.md`
   - ブック全体の統合Markdown
   - シート一覧
   - シート別セクション
   - 各シートの表・項目・入力規則
   - warningsへの参照

2. `sheets/<sheet-name>.md`
   - シート単位のMarkdown
   - 後続のRAG投入や人間レビューに使える形式

既存の `specification.md` は単一ファイル向けの出力名として残してもよいが、デモアプリでは `book_specification.md` を主成果物とする。

### 5.6 JSON出力

以下のJSONを出力する。

- `extracted.json`
  - ブック全体の抽出構造
  - シートごとの blocks / titles / validations / comments / warnings

JSONは後続のLLM整形・RAG投入・テストケース生成の入力として使える構造にする。

### 5.7 HTML preview

`preview.html` を出力し、ブラウザ上で確認できるようにする。

必須要素:

- ブック名
- シート別ナビゲーション
- 各シートの見出し
- 罫線ブロックの表表示
- セル座標の表示
- warnings への導線

目的は、変換結果を営業デモで説明しやすくし、人間が元Excelとの対応を確認できるようにすること。

### 5.8 評価レポート

`evaluation.md` を出力する。

含める内容:

- シート数
- 抽出した罫線ブロック数
- 結合セル見出し数
- 入力規則数
- コメント数
- warnings数
- 対象外・未対応要素
- 人手確認が必要な箇所

### 5.9 warnings表示

変換不能または不確かな情報は、成功扱いにせず warnings として明示する。

例:

- コメント付きセル
- 外部範囲参照の入力規則
- 非表示シート
- 空白が多い罫線ブロック
- 図形・画像・マクロなど未対応要素

## 6. 画面要件

### 6.1 画面一覧

初期デモは以下の3画面で構成する。

1. アップロード画面
2. 変換結果ダッシュボード
3. プレビュー・ダウンロード画面

### 6.2 アップロード画面

表示要素:

- サービス名
- 短い価値訴求
- ファイルアップロード領域
- 対応形式 `.xlsx`
- サンプルファイルで試すボタン

顧客向け画面なので、内部事情や開発中表現は出さない。

### 6.3 変換結果ダッシュボード

表示要素:

- ファイル名
- シート数
- シート分類一覧
- 抽出ブロック数
- warnings数
- 主要成果物へのリンク

この画面では、営業説明しやすいように数字をコンパクトに表示する。

### 6.4 プレビュー・ダウンロード画面

表示要素:

- シート別プレビュー
- Markdown表示
- warnings表示
- evaluation表示
- ダウンロードボタン

## 7. 非機能要件

### 7.1 セキュリティ

- 初期デモではアップロードファイルを永続保存しない。
- 変換処理後、必要に応じて一時ディレクトリを削除する。
- 外部LLM APIへファイル内容を送らない。初期デモは決定論的なローカル解析に限定する。
- 本番PoCでは、社内文書を扱うため、保存期間・アクセス制御・監査ログを別途設計する。

### 7.2 性能

- サンプルExcelは数秒以内に変換できること。
- 20MB程度のExcelで、営業デモ中に待てる時間内で処理できることを目標にする。
- 重いファイルでは進捗表示または処理中表示を出す。

### 7.3 使いやすさ

- デモ説明なしでも、アップロードから成果物確認まで進めるUIにする。
- 顧客向け文言は正式・事業向けにする。
- 「Excelの代替」ではなく「既存設計書をAI活用できる資産にする」と表現する。

---

## 8. 初期デモの対象外

以下は初期デモでは対象外とする。

- Word / PDF 変換
- `.xls` / `.xlsm` 対応
- 図形・画像・テキストボックスの完全抽出
- マクロ解析
- SharePoint / Box / Google Drive 連携
- ユーザー認証
- 組織別権限管理
- 本番RAG検索
- LLMによる高度な仕様再構成
- テストケース自動生成

ただし、営業上は次フェーズ拡張として説明可能にする。

---

## 9. 受け入れ条件

### 9.1 デモ成立条件

- サンプル `.xlsx` をアップロードできる。
- 複数シートを認識し、シート一覧を表示できる。
- シートごとのMarkdownを確認できる。
- ブック統合Markdownを確認できる。
- HTML previewでシート別の表構造とセル座標を確認できる。
- evaluationで抽出結果の件数を確認できる。
- warningsで人手確認箇所を確認できる。
- 成果物をダウンロードできる。

### 9.2 品質条件

- 変換できない要素を成功扱いしない。
- warnings に不確実性を明示する。
- 顧客向け画面に内部メモ・開発都合・ミラ口調・安い比較訴求を出さない。
- デモ用サンプル以外のファイル名や内容を画面に固定表示しない。

## 10. 推奨実装方針

初期デモは、既存の Python コンバータを変換エンジンとして使い、軽量なWebアプリを追加する。

推奨構成:

- Backend / converter: 既存 `jtc_excel_md` Python package
- Web app: Next.js または FastAPI + simple frontend
- 初期優先: ローカルで確実に動く営業デモ
- デプロイ: 後続で Cloudflare Pages / Render / Fly.io 等を検討

現時点のおすすめは、まず **ローカル完結のWebデモ** とすること。

理由:

1. 顧客文書を扱うため、初期から外部アップロード前提にしない方が安全。
2. 既存Python資産をそのまま使いやすい。
3. 営業デモでは「目の前でExcelを入れて成果物が出る」ことが最重要。
4. 認証・ストレージ・クラウド運用を後回しにできる。
